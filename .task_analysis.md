# ROOT CAUSE ANALYSIS — All 4 Issues

## Issue 1: "Agent Run ID: undefined" inside signal fusion

**Root cause file:** `operator/operator.js` line 167
**Root cause function:** `generateNewsletterContent()` → calling `fuseSignals()` without a `runId`

**Broken variable flow:**
1. `operator.js` `run()` begins → NO runId generated yet at pipeline level
2. `executePlan()` runs with `plan.steps` → calls `generateNewsletterContent()` at line 608
3. `generateNewsletterContent()` at line 167 calls:
   ```js
   const fusedSignals = await fuseSignals(CURRENT_DATE, editorialFrame);
   ```
   No third argument `{ runId }` is passed.
4. `fusion-engine.js` `fuseSignals()` at line 284:
   ```js
   export async function fuseSignals(currentDate, editorialFrame = null, options = {})
   ```
   Line 289: `const agentRunId = options.runId;` → `undefined`
5. Line 293: `console.log(\`Agent Run ID: ${agentRunId}\`);` → prints `"Agent Run ID: undefined"`
6. All subsequent `markAgentComplete("macro", agentRunId)` calls pass `undefined` as runId
7. Agent-orchestrator compares `undefined` against `state.runId = "test-run-123"` → mismatch warning

**Secondary problem:** line 175 generates a runId BUT it's too late — it's only used for `synthesizeInsights()`:
```js
const runId = `${JOB_NAME}-${CURRENT_DATE}-${Date.now()}`;
```
This runId is never passed to `fuseSignals()` or used as the pipeline-level runId.

**Evidence in code:**
- `operator/operator.js` line 167: `fuseSignals(CURRENT_DATE, editorialFrame)` — missing `{ runId }`
- `fusion-engine.js` line 289: `const agentRunId = options.runId;` — reads `undefined`

## Issue 2: agent-state.json contains stale runId "test-run-123"

**Root cause file:** `operator/operator.js` + `operator/core/agent-orchestrator.js`

**Root cause:** No component calls `initAgentRun()` at pipeline startup. The file `operator/logs/agent-state.json` survives across runs because:
1. `operator.js` `run()` never initializes agent state
2. The stale `agent-state.json` was left by a prior test run:
   ```json
   {"runId":"test-run-123","macro":"complete","gcc":"complete","risk":"complete","editor":"complete"}
   ```
3. `fuseSignals()` calls `markAgentComplete("macro", undefined)` which loads the stale file, sees `state.runId = "test-run-123"` !== `undefined`, logs a mismatch warning, but STILL writes to the stale state — mutating a stale record instead of creating a fresh one.

**Impact:** Each run mutates the stale "test-run-123" state, but the runId field is never updated to the current pipeline's ID. The barrier in `waitUntilAllAgentsComplete()` (line 155-158) then sees the mismatch and adopts the stale runId to prevent deadlock — creating a loop of stale state propagation.

## Issue 3: WEBHOOK_SECRET alignment

**Root cause:** Bootstrapping timing gap + field naming mismatch

**Node sender side (`operator/.env`):**
```env
WEBHOOK_SECRET=89e9d1671f9a13dbd3cbdc5fd90a2fdecaff7a5d635b81aa
```

**Node sender side (`operator/operator.js` pushToAppsScript):**
- Reads `process.env.WEBHOOK_SECRET` at line 377
- Injects it as `payload.auth_token` at line 384

**Apps Script receiver (`scripts/Code.gs` doPost):**
- Line 569: `var secretToken = PropertiesService.getScriptProperties().getProperty('WEBHOOK_SECRET')`
- Line 582: checks `contents.auth_token !== secretToken`

**The gap:**
1. The Node side sends `auth_token` in the payload
2. The Apps Script reads `WEBHOOK_SECRET` from PropertiesService
3. If the Apps Script was freshly deployed, PropertiesService is empty → `secretToken = null`
4. The auto-bootstrap exists in `doGet()` (line 500-502) but ONLY runs on GET requests
5. If the pipeline POSTs before someone visits the GET endpoint → `secretToken` is still `null`
6. Result: `{"status":"error","message":"Server misconfigured: WEBHOOK_SECRET not set"}`

**Additional issue — payload structure:**
The `pushToAppsScript` in operator.js sends the raw newsletter JSON (line 381-383):
```js
payload = JSON.parse(fs.readFileSync(absolutePath, "utf-8"));
payload.deliveryId = getDeliveryId(JOB_NAME);
payload.auth_token = webhookSecret;
```
This payload contains: `deliveryId`, `auth_token`, and newsletter fields like `date`, `title`, `sections`, `strategicInsights`, `scenarios`.

But the Apps Script `doPost()` checks for `contents.subject` and `contents.htmlBody` (line 618) to send email. The operator.js push does NOT send `subject` or `htmlBody` — only newsletter JSON. So even when auth passes, the email may not be sent because the expected fields are missing.

The delivery_log.txt shows "VALIDATED SUCCESS" for the most recent runs (meaning the HTTP 200 body was validated by `validateAppsScriptResponse()`). But the actual email behavior depends on what Apps Script returned — it may have matched the `"Payload received (no email — missing subject/htmlBody)"` path (code line 646-652).

## Issue 4: Truth evaluator git=false

**Root cause file:** `operator/core/truth-evaluator.js` lines 79-99

**Root cause function:** `verifyGit()` — incorrect success criteria

```js
function verifyGit() {
  const log = safeExec("git log -1 --format=%H");
  const status = safeExec("git status --porcelain");
  const commitHash = log.success ? log.output : null;
  const hasUncommitted = status.success && status.output.length > 0;
  const verified = log.success && !hasUncommitted;
  ...
}
```

**The failure mode:**
Git verification requires ZERO uncommitted changes (`!hasUncommitted`). But `evaluateTruth()` is called AFTER the pipeline's `executePlan()` step, during the post-pipeline phase that writes multiple files:
1. Feedback loop writes `operator/memory/newsletter-history.json` (line 66-72 of operator.js)
2. The optimizer writes to `operator/logs/optimization-log.json` and `operator/logs/feedback.json`
3. The truth evaluator itself writes to `operator/logs/truth-log.json`

All these modifications happen AFTER the git commit step inside `executePlan()`. So `git status --porcelain` will always return non-empty output by the time `verifyGit()` runs.

**Evidence in truth-log:**
```json
{"sources": {"git": false, "appsScript": true, "email": true}, "gitHash": "99ef689...", ...}
```
The `gitHash` IS populated (commit exists), but `verified` is false because of uncommitted changes.

**The real issue:** The git commit DID succeed (gitHash is non-null), but the verification requires zero uncommitted changes at evaluation time — which is impossible because evaluation writes files after the commit.
